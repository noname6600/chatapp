ChatApp Backend - Local Hybrid Runbook (Low-Lag Mode)

Purpose
- Run only infrastructure in Docker (PostgreSQL, Redis, Zookeeper, Kafka).
- Run backend services natively with Gradle bootRun for lower local lag and faster debug cycles.

Infrastructure coverage (includes service databases)
- Local hybrid now runs a dedicated PostgreSQL container per DB-backed service:
  - auth-db -> auth_service (auth-service)
  - user-db -> user_service (user-service)
  - chat-db -> chat_service (chat-service)
  - friendship-db -> friendship_service (friendship-service)
  - notification-db -> notification_service (notification-service)
- Each DB container is initialized with its own service-scoped user/password.

Database execution model (important)
- There are five PostgreSQL containers in local hybrid (one per DB-backed service).
- Isolation is enforced by container boundary plus service-scoped DB credentials.
- There is no shared app Postgres container in this local compose mode.

Scope Rules
- In hybrid mode, DO NOT run backend application containers from docker-compose.local.yml.
- Use Docker only for infrastructure services.

Working Directory
- Run all commands from chatappBE unless stated otherwise.

Preflight
1) Required tools
- docker --version
- docker compose version
- Java 21+ and Gradle wrapper available

2) Required files
- .env.local exists and contains local values
- .env.local includes service-scoped DB variables (`*_DATABASE_NAME`, `*_DATABASE_USER`, `*_DATABASE_PASSWORD`)

3) Recommended shell split
- Terminal 1: Docker infrastructure
- Terminal 2: auth-service
- Terminal 3+: remaining services

Multi-Service Startup (Recommended)
Use the convenience scripts to open all services in separate windows with one command.
IMPORTANT: Docker infra must be running first (see "Infrastructure Startup" section).

From PowerShell (chatappBE root):
  .\scripts\start-services-local.ps1             # starts all 8 services
  .\scripts\start-services-local.ps1 -Auth -Gateway  # starts only auth + gateway
  .\scripts\start-services-local.ps1 -Auth -User -Chat -Gateway  # starts a custom subset

From cmd (chatappBE root):
  scripts\start-services-local.bat               # starts all 8 services
  scripts\start-services-local.bat -Auth -Gateway

Available service switches: -Auth -User -Chat -Presence -Friendship -Notification -Upload -Gateway

If PowerShell execution policy blocks the script:
  powershell -ExecutionPolicy Bypass -File scripts\start-services-local.ps1

1) Start infrastructure (auth-db, user-db, chat-db, friendship-db, notification-db, redis, zookeeper, kafka)

5) Kill stale Java listeners on service ports (PowerShell)
$ports = 8080,8081,8082,8083,8084,8085,8086,8088
foreach ($p in $ports) {
  $pids = Get-NetTCPConnection -LocalPort $p -State Listen -ErrorAction SilentlyContinue |
    Select-Object -ExpandProperty OwningProcess -Unique
  foreach ($pid in $pids) {
    Stop-Process -Id $pid -Force
  }
}

.env.local Caveats
- RESEND_API_KEY and RESEND_FROM must be provided (env vars) for auth email delivery via Resend.
- GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET defaults are provided in `auth-service/application-local.yaml` for local development; override if needed.
- CLOUDINARY_CLOUD_NAME, CLOUDINARY_API_KEY, CLOUDINARY_API_SECRET defaults are provided in local profiles for upload/user/chat flows; override if needed.
- FRONTEND_URL should be http://localhost:5173 for local browser flow.
- DB-backed services use service-scoped credentials (`*_DATABASE_USER` and `*_DATABASE_PASSWORD`) and dedicated DB containers in local compose.
- presence-service and upload-service are intentionally out of scope for PostgreSQL credentials in this change.
- If any required value is missing, the related service may fail on startup.

Troubleshooting (cmd)
- `'gradlew.bat' is not recognized`: you are not in `chatappBE`; run `cd /d d:\Work\PET\chatappPET\chatapp\chatappBE`.
- `Could not resolve placeholder ...`: add or override the missing key in service `application-local.yaml`, or set it inline in cmd before bootRun.
- `Unknown command-line option '--spring.profiles.active'`: in cmd, use `--args=--spring.profiles.active=local` (without nested quotes).
- `Connection refused` to Postgres/Redis/Kafka: start infra first with docker compose local commands from this runbook.

Troubleshooting: Gateway Startup Failures
Common issue: Gateway fails to start with `Failed to generate bean name for imported class` or `Could not find class ...` errors.
- Root cause: Spring Cloud Commons version mismatch with backend Spring Boot version.
- Solution: All services now run on Spring Boot **3.5.6** (was 4.0.1). This ensures Spring Cloud Gateway artifacts are compatible.
- If you see class-resolution bootstrap errors for Spring Cloud auto-config classes:
  1. Verify you are using the current backend code (pull latest changes).
  2. Check that all service `build.gradle` files use Spring Boot 3.5.6 plugin (no longer 4.0.1).
  3. Re-run `.\gradlew.bat :gateway-service:bootRun --args=--spring.profiles.active=local`.
- If command syntax is the issue (e.g., "Unknown command-line option"), check `Troubleshooting (cmd)` section above.

Auth Provisioning Rollout / Rollback Notes (Login Error + Incomplete Account)
Rollout steps
1) Deploy backend modules that include `INCOMPLETE_ACCOUNT` and deterministic invalid-credentials messaging:
  - common/common-web
  - auth-service
2) Deploy frontend changes that:
  - parse nested backend error envelopes (`error.code`, `error.message`)
  - await login bootstrap before navigating
  - retry user profile bootstrap briefly before controlled logout
3) Verify in local/staging:
  - Wrong password shows "Invalid credentials"
  - First-login/profile-lag shows "Account setup incomplete. Please try again in a few seconds."
  - No refresh-loop behavior for `INCOMPLETE_ACCOUNT`

Rollback steps
1) Revert frontend first if users report unexpected login bootstrap errors.
2) Revert auth-service/common-web together if backend clients are not yet compatible with `INCOMPLETE_ACCOUNT` code.
3) After rollback, confirm:
  - auth login/register endpoints return previous behavior
  - frontend login remains functional with generic auth error messages
